Claude Code Compliance Audit — Not-Started Features

Date: 2026-05-18  |  Auditor: Claude Code (Opus 4.7) interactive session, no claude -p  |  Scope: 16 not-started features in docs/features-v2.md

Methodology

This audit is a slice of the broader F12 (docs/audit-claude-code-compliance) feature, which targets the entire pipelinekit repository. This slice covers only the not-started feature entries in docs/features-v2.md: 5 pre-existing entries (F8–F12) plus 11 new entries promoted in the same session that produced this report. The full repo-wide F12 audit remains open.

Each feature is evaluated across nine dimensions:

  1. Skill structure — frontmatter schema (name, description, allowed-tools, model, etc.) per Claude Code skills docs.
  2. Agent definitions — frontmatter (name, description, tools, model, permissionMode) per sub-agents docs.
  3. Hook signatures & triggers — event names, JSON output schema, settings.json registration per hooks docs.
  4. MCP server configuration — .mcp.json schema fields per MCP docs.
  5. Plan vs. implementation alignment — does the Description match the Constraints (no overreach / underreach)?
  6. /pipeline autonomy — does the feature preserve F6's "Phase 0 charter → merge runs fully autonomously without further prompts" outcome?
  7. Commit hygiene — conventional commits, no AI attribution.
  8. Branch / PR workflow — <type>/<slug> naming, squash-merge protocol.
  9. Native Claude Code overlap — is the planned functionality already available natively in Claude Code or the Agent SDK? If yes, flag and recommend re-scoping to wrap/extend the native surface. EXCEPTION: pipelinekit's flat-file + agentmemory memory system is intentionally superior to Claude Code's native memory; the memory dimension is exempted from overlap flagging per explicit user direction.

Per-cell verdicts:

  • PASS No issue.
  • NB Non-blocking finding; fix advisable when the feature is picked up.
  • B Blocking finding; the feature design needs revision before pickup.
  • N/A Dimension does not apply to this feature.

Detailed Findings

F8 feat/blacksmith-ci-default

5 PASS · 0 NB · 0 B · 4 N/A

No findings. Blacksmith is a third-party CI runner; the feature is internal to .github/workflows/* and /create-plan guard logic with no Claude Code overlap. The BLACKSMITH_RUNNER env-var default is sensible.

F9 feat/pipeline-mobile-notifications

5 PASS · 1 NB · 1 B · 2 N/A

BDimension 9 (Native overlap) — Blocking.

Claude Code 2.1.110 ships a native PushNotification tool that Claude can use directly to send mobile push notifications when Remote Control + "Push when Claude decides" config are enabled. Claude Code 2.1.121, 2.1.126, 2.1.129 add Channels (--channels, channelsEnabled: true) for inbound push from Telegram / Discord / iMessage / webhooks, and Remote Control evolved through 2.1.122 / 2.1.129. The pipelinekit feature acknowledges these in its Description ("Investigate Claude Code's PushNotification tool / Stop hook / Notification hook surface") but its Constraints describe building a custom rate-limited notification surface ("Notifications are rate-limited (no more than 1 per minute per feature)", "User can disable per-run via --no-notifications or globally via ~/.claude/config/notifications.json") that duplicates infrastructure Claude Code already provides.

Sources: CHANGELOG.md (2.1.110 / 2.1.121 / 2.1.122 / 2.1.126 / 2.1.129), Claude Code overview ("Remote Control" + "Channels").

Recommended fix: Re-scope the feature to integrate with native surfaces rather than build alongside them. Concretely: (a) the pipeline emits notifications via the native PushNotification tool when invoked from a Claude Code session; (b) external triggers (e.g., budget breaches) use the Notification hook to surface terminal escape sequences; (c) the rate-limit + opt-out config can be a thin wrapper around Channels' channelsEnabled rather than a parallel config file. Drop ~/.claude/config/notifications.json. Update Description to specify which native surface handles each event-type row.

NBDimension 3 (Hook signatures) — Non-blocking.

The Constraints mention wiring "the pipeline's halt/question paths into it" but don't specify which hook event fires the notification. The current hook event list is: SessionStart, UserPromptSubmit, Stop, StopFailure, PreToolUse, PostToolUse, PermissionRequest, PermissionDenied, SubagentStop, Notification, etc.

Source: Hooks reference.

Recommended fix: Specify a mapping: Stop hook for end-of-feature, PermissionRequest for question/auth prompts, Notification for budget/error events, PreCompact for context-fill warnings.

F10 feat/integrate-openhuman

4 PASS · 4 NB · 0 B · 1 N/A

NBDimension 9 (Native overlap) — Non-blocking.

Claude Code provides several native surfaces that overlap with parts of openhuman's human-in-the-loop story: PermissionRequest / PermissionDenied hooks for confirmation gates, plan mode's ExitPlanMode approval dialog, and the Notification hook for surfacing pending decisions to the user. EnterWorktree / ExitWorktree native tools (added/improved in 2.1.101 and 2.1.143 with worktree.bgIsolation / worktree.baseRef settings) cover worktree management that openhuman might otherwise reinvent.

Sources: Hooks reference, CHANGELOG.md (2.1.101, 2.1.143).

Recommended fix: Before vendoring tinyhumansai/openhuman, audit which openhuman capabilities are genuinely additive (e.g., out-of-band webhook approvals, structured human-review templates) vs. duplicates of native PermissionRequest + Notification. Trim the vendored surface to the additive parts.

NBDimension 6 (/pipeline autonomy) — Non-blocking.

Constraints say "configurable timeout (default 60 min); on timeout, fail-safe path is abort, not auto-approve". A 60-minute wait inside an autonomous pipeline run breaks F6's autonomy promise ("Phase 0 charter → merge runs fully autonomously"). The default is too long to be considered "autonomous" — a pipeline started at 9 AM might still be waiting at 10 AM.

Source: F6 outcome in docs/features-v2.md.

Recommended fix: Either (a) default timeout = 0 (no wait) with explicit opt-in via --human-review <minutes>, or (b) move human-review to an async channel (background queue) so the pipeline doesn't block. Document the trade-off explicitly.

NBDimension 1 (Skill structure) — Non-blocking.

Constraints mention "wire as a skill/agent under claude/skills/openhuman/ (or whatever the upstream layout dictates)". The skill frontmatter (name, description, allowed-tools, model, disable-model-invocation) is not specified.

Source: Skills frontmatter reference.

Recommended fix: Specify frontmatter in advance: name: openhuman, description: Pause for out-of-band human approval before destructive actions, disable-model-invocation: true (user-triggered only), allowed-tools: Bash(curl *).

NBDimension 3 (Hook signatures) — Non-blocking.

"Triggers configurable per-feature: by default human-in-the-loop fires only at squash-merge step" — which hook does this register against? PreToolUse matching Bash(git merge *)? PermissionRequest for the squash command? The Constraints don't say.

Source: Hooks reference (matcher patterns, PreToolUse with Bash(...)).

Recommended fix: Specify hook config in plan: PreToolUse with matcher Bash and an if clause Bash(git merge --squash *), type http or command pointing at the openhuman handler.

F11 feat/docs-vs-documentation-split

6 PASS · 1 NB · 0 B · 2 N/A

NBDimension 1 (Skill structure) — Non-blocking.

Constraints mention "Update /docs-writer agent to emit .html to documentation/ instead of .md." The naming convention /docs-writer (slash-prefixed) suggests it's a skill or command, not an agent. Claude Code's docs distinguish: skills live under .claude/skills/ with SKILL.md; agents live under .claude/agents/. The feature should clarify which it is.

Source: Skills vs. Sub-agents.

Recommended fix: Resolve the ambiguity in the plan. If it's a skill, the plan should target claude/skills/docs-writer/SKILL.md; if an agent, claude/agents/docs-writer.md. Pick one.

F12 docs/audit-claude-code-compliance

7 PASS · 1 NB · 0 B · 1 N/A

NBDimension 9 (Native overlap) — Non-blocking.

Claude Code ships built-in /review and /security-review commands (accessible via the Skill tool) that perform code-review-style audits. F12's scope is broader (compliance audit across skill/agent/hook/MCP/pipeline structure, not just code review), so the overlap is partial — but the relationship deserves explicit documentation so users understand when to reach for native /review vs. the pipelinekit audit.

Source: Skills (bundled commands: /init, /review, /security-review are accessible through the Skill tool).

Recommended fix: Add a "Relationship to native /review and /security-review" section to F12's audit output that explains: native /review reviews a diff; native /security-review reviews security on pending changes; pipelinekit audit covers compliance with Claude Code conventions across the repo (a meta-level audit, not per-PR).

N1 feat/pipeline-renew-charter-revalidation

6 PASS · 2 NB · 0 B · 1 N/A

NBDimension 6 (/pipeline autonomy) — Non-blocking.

Constraints specify "Surface the drift report via AskUserQuestion so the user explicitly accepts/rejects each drifted line before features are regenerated." This is mid-pipeline prompting, which conflicts with F6's "Phase 0 charter → merge fully autonomously" outcome. --renew by definition implies the user wants to re-validate, so a prompt may be desired here — but the feature should clarify.

Source: F6 Run Log in docs/features-v2.md; Skills reference notes AskUserQuestion is interactive-only and won't work in subagent context.

Recommended fix: Add a Constraint: "AskUserQuestion is only invoked when --renew runs in interactive mode; when run via --renew --auto, the orchestrator accepts all 'drifted' lines without prompting and notes the accepted set in the resulting feature file."

NBDimension 1 (Skill structure) — Non-blocking.

The feature extends claude/skills/pipeline/SKILL.md but doesn't specify whether the new flag handling lives in the main SKILL.md or a separate reference.md supporting file (per Skills' supporting-files pattern). Adding flag logic inline grows SKILL.md and risks the "skill descriptions cut short" warning when the description budget overflows.

Source: Skills § "Add supporting files" and § "Skill descriptions are cut short".

Recommended fix: Land the new flag logic in claude/skills/pipeline/reference.md (existing pattern), referenced from SKILL.md with a one-line pointer.

N4 feat/plan-decomposition-compliance

6 PASS · 1 NB · 0 B · 2 N/A

NBDimension 9 (Native overlap) — Non-blocking.

Claude Code has native plan mode (entered via ExitPlanMode tool) and a /goal command (added 2.1.139) that "sets a completion condition and Claude keeps working across turns until it's met". Neither directly implements the three-axis decomposition gate (TDD pairing + parallel zero-overlap + vertical slices), but both are adjacent surfaces. The feature documentation should explain the relationship so users don't conflate plan-mode plans with pipelinekit's /create-plan output.

Source: CHANGELOG.md (2.1.139), Claude Code overview (plan mode references).

Recommended fix: Add a paragraph to the feature's Description distinguishing: (a) Claude Code plan mode = pre-execution preview of intended changes for user approval; (b) /goal = self-pacing loop; (c) decomposition-compliance gate = validation of /create-plan output against three axes. These are non-overlapping but complementary.

N5 feat/sandbox-orchestrate-wrapper

6 PASS · 1 NB · 0 B · 2 N/A

NBDimension 9 (Native overlap) — Non-blocking.

Claude Code 2.1.143 added worktree.bgIsolation and worktree.baseRef settings to the EnterWorktree tool, providing native git-worktree-based isolation for background workers. Pipelinekit's sandbox abstraction (podman/docker/worktree-only) operates at the container level (stricter isolation), but for the worktree-only provider there's now overlap with native worktree isolation.

Source: CHANGELOG.md (2.1.143).

Recommended fix: Add to Constraints: "When SANDBOX_PROVIDER=worktree-only, delegate to native EnterWorktree with worktree.bgIsolation=true rather than implement a parallel worktree path. The pipelinekit-specific code paths remain only for the podman and docker providers."

Memory dimension exemption

Per explicit user direction at audit-commissioning time, the "native Claude Code overlap" dimension explicitly excludes any finding tagged against pipelinekit's memory system. Claude Code ships a native auto memory feature with CLAUDE.md-based persistence; pipelinekit's flat-file (~/.claude/memory/ indexed by MEMORY.md with recency-weighted confidence decay) plus the vendored agentmemory retrieval layer is considered intentionally superior. Any overlap signal between pipelinekit memory and native Claude Code memory is suppressed and does NOT appear in the findings above. This audit checks all other dimensions normally.
