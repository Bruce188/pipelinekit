Executive summary

Pipelinekit is broadly compliant with current Claude Code (2.1.144) standards. Skill and agent frontmatter conform to the documented schemas; hook registrations cover the major event types; the /pipeline autonomous loop respects the documented autonomous-mode contract; commit hygiene is clean (no AI-attribution leakage in any of the last 50 main commits).

The audit found 0 blocking, 9 non-blocking, and 7 nit-level findings across the 9 audit dimensions. Most non-blocking findings concern inconsistent allowed-tools / tools separator conventions across the skill and agent fleets (the docs accept space-separated or YAML list; pipelinekit's house style is comma-separated, which works empirically but isn't in the canonical spec). Two findings concern recently-shipped Claude Code surfaces pipelinekit hasn't yet picked up (worktree.bgIsolation, worktree.baseRef from 2.1.143; /goal from 2.1.139). Two findings concern the inert agentmemory entry in .mcp.json.template still using a _pending_real_mcp_shape placeholder despite @agentmemory/mcp being live in user-level MCP config.

Memory exemption honored: per the F23 spec, the pipelinekit memory system (flat-file ~/.claude/memory/ + vendored agentmemory retrieval) is exempt from native-overlap flagging; it is intentionally a superset of native auto-memory.

Relationship to native /review and /security-review

Claude Code ships two built-in review-style skills that overlap conceptually with this audit but operate at a different scope:

SurfaceScopeTriggerOutput
Native /review A single PR diff, evaluated for quality, style, and bugs. Per-PR code review. User-invoked, typically before merging a PR. Inline findings list scoped to the diff.
Native /security-review Pending changes on the current branch, evaluated for security regressions (auth bypass, injection, secrets, etc.). User-invoked, typically before pushing. Security-focused findings list on the diff.
Pipelinekit /review (multi-agent) A single pipeline-feature diff, reviewed by 2–6 specialized agents in parallel (test-engineer, security-auditor, karpathy-reviewer, spec-tracer, code-reviewer, etc.). Dispatched by the /pipeline orchestrator at Step 5.6 of each feature. Versioned docs/review-vN.md consumed by routing logic (Path A/B/M/N).
This audit (docs/audit-claude-code-compliance) Meta-level compliance check across the whole repo. Skill / agent / hook / MCP / pipeline / commit / branch / native-overlap structure — NOT a per-diff review. Manually invoked (forbids claude -p; runs in interactive Claude Code). This HTML report.

When to use which:

  • Native /review → before merging any PR for a quick automated read-through of the diff.
  • Native /security-review → before pushing anything security-sensitive (auth, credentials, parsers).
  • Pipelinekit /review → fires automatically during /pipeline runs; you don't invoke it manually.
  • This audit → quarterly or after major Claude Code releases, to detect drift between pipelinekit and the evolving Claude Code spec.

D1 — Skill structure (frontmatter)

Inventory: 44 skills, all with SKILL.md present. 42 PASS 2 NB

Canonical schema (per docs/en/skills#frontmatter-reference): all fields optional; only description recommended. Allowed fields: name, description, when_to_use, argument-hint, arguments, disable-model-invocation, user-invocable, allowed-tools, model, effort, context, agent, hooks, paths, shell.

NB-D1-1 — allowed-tools uses comma separators repo-wide; spec says space-separated string or YAML list

Spec says (docs): "Accepts a space-separated string or a YAML list."
Repo state: 22 of 44 skills declare allowed-tools using commas, e.g. analyze: allowed-tools: Read, Glob, Grep, Bash, Write, AskUserQuestion, WebSearch, WebFetch, mcp__local-rag, mcp__context7. A space-separated reading would treat the entire string as one tool name; in practice Claude Code seems to tokenize on commas too, but this is not in the documented contract.
Severity: non-blocking. Behavior is empirically correct (every skill in production works), but the convention diverges from the documented format and could break if the parser tightens. Recommended fix: normalize to YAML list form across the fleet: 
allowed-tools:
  - Read
  - Glob
  - Grep
or to space-separated string per the docs. Bundle into chore/post-merge-cleanup.

NB-D1-2 — test-hooks skill uses JSON-array-string for paths

Repo state: claude/skills/test-hooks/SKILL.md has paths: ["**/.claude/hooks/**"]. The spec lists comma-separated string OR YAML list; JSON-inline-array-as-string is undefined.
Severity: non-blocking (string parse likely treats the brackets as part of the path glob, which would fail silently rather than matching anything). Recommended fix: change to paths: "**/.claude/hooks/**" or YAML list form.

Other observations (no findings)

  • claude-md-enhancer declares disable-model-invocation: false — redundant but harmless (default is false).
  • simplify uses a YAML-quoted description (`"…"`); others don't. Cosmetic.
  • Several reference-content skills (tdd, ios, playwright, expo, financial-data-analyst) intentionally omit allowed-tools to inherit session defaults — correct per the docs ("All fields are optional").
  • Descriptions are well within the 1,536-char cap; no truncation risk in the listing.

D3 — Hook signatures & triggers

Inventory: 24 hook scripts; 13 event types registered in settings.json (PreToolUse, PostToolUse, PostToolUseFailure, PreCompact, PostCompact, SessionStart, SessionEnd, Stop, SubagentStart, SubagentStop, UserPromptSubmit, Notification, TaskCompleted). 22 PASS 2 NB

NB-D3-1 — Missing args: string[] exec form for hooks (2.1.139+)

Spec says (CHANGELOG 2.1.139): hooks can use args: string[] exec form for shell-injection safety. The exec form spawns the command directly without shell tokenization.
Repo state: All pipelinekit hooks in settings.json use the legacy command: "..." shell form. Spot-check of ~/.claude/settings.json shows no args arrays anywhere.
Severity: non-blocking. Pipelinekit hooks don't accept untrusted input (all are local scripts), so the injection risk is minimal. But the exec form is the new recommended pattern for any hook taking variable input. Recommended fix: migrate PreToolUse hooks (especially block-stage-sensitive.sh, block-dangerous-commands.sh, env-scrub.py) to the exec form where they interpolate file paths or args. Bundle.

NB-D3-2 — Missing terminalSequence usage for desktop notifications (2.1.141+)

Spec says (docs/en/hooks): hooks can emit OSC desktop-notification escape sequences via the terminalSequence field in the JSON output, replacing the older /dev/tty write pattern (which broke in 2.1.139 when hooks lost the controlling terminal).
Repo state: claude/hooks/notify-emit.sh uses an OSC 777 sequence pattern (already correct shape per CHANGELOG 2.1.141), but emits it via stdout text rather than via the terminalSequence JSON field. The pipelinekit /pipeline reference describes this as the "OSC 777 terminalSequence path" — the doc references the right surface but the script implementation predates the JSON-field landing.
Severity: non-blocking. Empirically the notifications fire on Windows Terminal / iTerm2 / Ghostty via the legacy raw-stdout path. Recommended fix: wrap the OSC sequence in the {"terminalSequence": "..."} JSON output for forward compatibility. ~10 LOC bash change.

Other observations (no findings)

  • All 24 hook scripts are executable (+x) and use POSIX #! shebangs.
  • Exit code conventions are correct: exit 2 = blocking (used in block-push-main.sh, block-stage-sensitive.sh), exit 0 = pass.
  • Hook test harness coverage is solid (claude/hooks/tests/test_*.sh, all green per 2026-05-19 test sweep).

D4 — MCP server configuration

Inventory: /home/bruce/pipelinekit/.mcp.json.template only (no active project-scope .mcp.json). User-level MCP config provides the actual servers at runtime. 3 PASS 2 NB

NB-D4-1 — Template's agentmemory entry sits in mcpServers block without command/args

Spec says (docs/en/mcp): every server in mcpServers needs at minimum a type and the type-specific transport fields (command + args for stdio).
Repo state: .mcp.json.template declares: 
"agentmemory": {
  "type": "stdio",
  "_pending_real_mcp_shape": "Upstream rohitg00/agentmemory has not shipped an MCP server binary as of F2. Replace this key with the real { command, args } block when upstream lands.",
  "env": { "AGENTMEMORY_STORE_MODE": "", "AGENTMEMORY_DATA_DIR": "" }
}
This entry has no command or args. If a project copies the template verbatim to .mcp.json and tries to start, Claude Code's MCP loader will reject or skip the entry (depending on version).
Severity: non-blocking. The template is documented as a starting-point that the user customizes. The _pending_real_mcp_shape key is explicitly an anti-footgun. Recommended fix: move agentmemory out of mcpServers into a sibling _pending_mcpServers namespace (matching the existing _mobile_mcpServers / _claude_context_mcpServers pattern), so a verbatim copy doesn't try to register a broken server. Also: now that @agentmemory/mcp (Zilliz-published) IS shipping (verified: the user-level config spawns it as an active MCP), update the placeholder block to the real shape.

NB-D4-2 — serena uses unpinned git+https://...@main

Template state: 
"serena": {
  "args": ["--from", "git+https://github.com/oraios/serena@main", "serena-mcp-server", ...]
}
The template's own _security_note acknowledges this: "git+https without a pinned ref allows upstream takeover to execute arbitrary code via uvx. Replace 'main' with a known-good commit hash."
Severity: non-blocking — the comment already documents the risk and instructs users to pin. Recommended fix: the template ships a placeholder SHA (or version tag like v0.x.y) so the "do this before sandbox use" step has a concrete starting point rather than asking the user to invent one. Same applies to ios-simulator-mcp.

Other observations

  • Template is well-commented with security notes inline — exemplary documentation.
  • Three opt-in namespaces (_mobile_mcpServers, _claude_context_mcpServers) follow a consistent pattern.
  • User-level ~/.claude/.mcp.json (if it exists) was not audited — out of scope.

NB-D9-1 — worktree.bgIsolation + worktree.baseRef (2.1.143) not yet referenced

Spec (CHANGELOG 2.1.143): two new worktree settings landed. worktree.bgIsolation: "none" opts out of background isolation. worktree.baseRef controls base ref (fresh | head).
Repo state: features-v2.md F16 description references the OLD worktree.bgIsolation=true (boolean) form, but the actual 2.1.143 syntax is a string ("none" or omitted=default). The shipped F16 code in orchestrate.sh doesn't use either setting yet — they're just mentioned in the feature description.
Severity: non-blocking. Pipelinekit's worktree usage works without these settings. Recommended fix: a follow-up feature could wire worktree.bgIsolation into SandboxProvider.sh for users who want to disable background isolation. Update the F16 description's syntax reference at the same time.

NB-D9-2 — /goal command (2.1.139) not referenced in /pipeline docs

Spec: /goal sets a session-level completion condition that the model evaluates each turn.
Repo state: /pipeline implements its own goal-tracking via docs/pipeline-state.md + Run Log entries — independent of /goal. The features-v2.md plan-decomposition-compliance entry (F15) mentions distinguishing from "native plan mode + /goal command" but no integration exists.
Severity: non-blocking. /pipeline's goal-tracking is feature-bound, not session-bound; the two surfaces serve different scopes. Recommendation: document the difference in docs/pipeline.md § Charter Mode so users know when to use each. No code change needed.

Nit-level (5 — no action recommended)

  • N-D9-1: /skill-creator bundled skill (per docs) overlaps slightly with pipelinekit's /write-a-skill (vendored from upstream). Both are present; users can pick. No action — vendored version is intentional.
  • N-D9-2: /simplify bundled skill is identical to pipelinekit's /simplify (both vendored from Anthropic). No action — vendored deliberately for offline support.
  • N-D9-3: --ultrareview subcommand (2.1.120) exists natively; pipelinekit's multi-agent /review --teams is functionally adjacent but more configurable. Both useful in different contexts.
  • N-D9-4: /cost + /stats merged into /usage (2.1.119). Pipelinekit's cost_log.py emits its own event log; users can cross-reference with /usage for the session-level view.
  • N-D9-5: ${CLAUDE_EFFORT} skill variable (2.1.120) could be used by pipelinekit skills to gate verbose vs concise output. Not currently used. Cosmetic.
